Architecture

Microservices communicating asynchronously through events

All traffic flows through Spring Cloud Gateway. Routes requests to auth-service, product-service, and inventory-service based on path. Handles CORS and acts as a transparent proxy without inspecting JWT payloads.

Reverse Proxy CORS Spring Cloud Gateway

Handles user registration, login, and JWT refresh. Signs tokens with RS256 using a 2048-bit RSA key pair. Exposes a public key and JWKS endpoint for downstream services to validate tokens locally.

REST API RS256 JWT BCrypt Event Publisher

Manages product catalog with full CRUD operations. Validates JWTs locally via Spring Security Resource Server (no runtime calls to auth-service). Publishes domain events to RabbitMQ on state changes.

REST API RS256 Validation Event Publisher

Maintains stock levels for all products. Consumes product events from RabbitMQ to synchronize inventory in real-time. Validates JWTs locally with RS256 public key. Each service owns its dedicated database.

Event Consumer Stock Management Database per Service

All services expose metrics via /actuator/prometheus (Spring Boot Micrometer), HikariCP connection pools, and RabbitMQ broker stats. cAdvisor collects container-level CPU, memory, network, and disk for every running service. PostgreSQL exporters expose database health per instance. Prometheus scrapes every 10s and Grafana visualizes with pre-built dashboards.

Development Roadmap

Incremental sprints building toward a complete platform

  • Project structure & development workflows
  • Angular frontend with routing & authentication
  • Product Service specification
  • Inventory Service specification
  • Dedicated auth-service with RS256 JWT + API Gateway
  • Product CRUD with PostgreSQL (JWT-protected)
  • RabbitMQ integration & ProductCreated event
  • Inventory consumer & stock synchronization
  • ProductUpdated & ProductDeleted events
  • Product listing, detail, create & edit pages
  • Monitoring & observability (Prometheus, Grafana, cAdvisor)
  • Order Service with dedicated database
  • ORDER_CREATED event & inventory reservation
  • Shopping cart & checkout flow
  • Retry & Dead Letter Queue patterns
  • Error handling & event replay
  • Alerts & dashboard refinement

Documentation

Specifications, architecture decisions, and API references

Every feature starts as a specification in specs/. Architecture, services, events, and database schemas are defined before any implementation begins. Six ADRs document every design decision.

The specs/ directory contains PRD, service contracts, event schemas, database designs, frontend requirements, and generated implementation tasks. All specs are version-controlled alongside the code.

All architectural decisions are documented as ADRs in specs/decisions/ — from RabbitMQ topic exchange design to database-per-service strategy, JWT authentication, Docker Compose orchestration, and Flyway migrations.
