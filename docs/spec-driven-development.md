# Spec-Driven Development (SDD)

## Overview

This project follows a Spec-Driven Development (SDD) approach.

Instead of writing code first and documenting later, the workflow is:

```text
Idea
  ↓
Specification
  ↓
Architecture Review
  ↓
Implementation
  ↓
Testing
  ↓
Documentation Update
```

Every feature, service, event, API endpoint, and database change must be described in a specification before implementation.

The objective is to:

* Improve code quality
* Reduce implementation ambiguity
* Generate better results with AI coding assistants
* Maintain architectural consistency
* Document business decisions

---

# Why SDD?

Traditional development often follows:

```text
Code First
    ↓
Documentation Later
```

This frequently leads to:

* Missing documentation
* Inconsistent APIs
* Unclear requirements
* Tight coupling between services

With SDD:

```text
Specification First
        ↓
Implementation
```

The specification becomes the single source of truth.

# Project Structure

```text
specs/
├── prd/           # Product Requirements Documents
├── architecture/  # System architecture, service interactions, infrastructure
├── services/      # Service responsibilities, APIs, business rules, security
├── tech-specs/    # Technical specifications
├── events/        # Event contracts, producers, consumers, payloads
├── database/      # Database ownership, entities, relationships
├── frontend/      # Pages, user flows, API integrations, UI requirements
├── decisions/     # Architecture Decision Records (ADRs)
├── tasks/         # Implementation tasks
└── reviews/       # Review reports
```

Each directory contains specifications for a specific domain. Tasks and reviews are generated by workflow phases.

# Architecture Decision Records (ADR)

Location:

```text
specs/decisions/
```

Purpose:

Document important architectural decisions.

Examples:

```text
specs/decisions/
├── adr-001-rabbitmq.md
├── adr-002-postgresql.md
└── adr-003-jwt-authentication.md
```

Example:

```md
# ADR-001

## Decision

Use RabbitMQ as the messaging broker.

## Reason

- Excellent Spring Boot integration
- Supports retries and DLQ
- Widely used in enterprise systems

## Consequences

- Additional infrastructure
- Event contract management
```

# Development Workflow

This project follows a 5-phase Spec-Driven Development cycle.
Each phase has a corresponding workflow file under workflows/.

## Phase 1 — Planning

Create or update specifications.

Refer to: workflows/planning.md

Examples:

```text
specs/architecture/
specs/services/
specs/events/
specs/database/
specs/frontend/
specs/decisions/
```

---

## Phase 2 — Refinement

Review and improve specifications before task generation.

Refer to: workflows/refinement.md

Questions:

* Does this affect RabbitMQ events?
* Does this require database changes?
* Does this introduce new dependencies?

---

## Phase 3 — Task Generation

Decompose specifications into executable tasks.

Refer to: workflows/task-generation.md

Output:

```text
specs/tasks/
```

---

## Phase 4 — Implementation

Implement a single task with tests.

Refer to: workflows/implementation.md

Only after specifications are approved.

---

## Phase 5 — Review

Validate implementation against specifications and standards.

Refer to: workflows/review.md

Output:

```text
specs/reviews/
```

# Using AI Assistants

Before generating code, always provide the relevant specifications.

Examples:

```text
Read specs/services/product-service.md and implement the Product Service.
```

```text
Read specs/events/product-created.md and implement the RabbitMQ producer and consumer.
```

```text
Read specs/database/postgres.md and create Flyway migrations.
```

This significantly improves AI-generated code quality and consistency.

# SDD Principles

The project follows these principles:

1. Specification First
2. Documentation as Code
3. Architecture Before Implementation
4. Explicit Event Contracts
5. Database Ownership per Service
6. Incremental Evolution
7. Traceable Decisions via ADRs

---

# Goal

The purpose of SDD is not to create bureaucracy.

The purpose is to ensure that every architectural decision, API contract, event definition, and business rule is documented before implementation, making the project easier to maintain, scale, and evolve over time.
